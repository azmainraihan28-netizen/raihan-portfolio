# Day 21 of 30 — LinkedIn Post

## 🐛 Workflow Topic
**Bug Report Triage: GitHub Issue → AI Severity Score → Auto-Label + Assign + Slack Alert**

---

## 📌 Pain Points Identified
1. Engineering teams waste 30–60 min/day just reading and manually categorizing GitHub issues
2. Critical P0 bugs sit unnoticed for hours because they look like regular tickets
3. Issues get assigned to the wrong team — backend bug goes to frontend dev, wasting everyone's time
4. No automatic escalation — a system-down bug gets the same silence as a typo fix
5. Growing teams with 50+ open issues at a time have zero triage system — chaos mode

## 💡 Solution (Short)
n8n workflow that fires the moment a GitHub issue is opened → AI reads title + description → assigns P0/P1/P2/P3 severity → labels it → assigns to the right team area → posts a triage comment on the issue → fires a Slack alert if it's critical. Zero manual work.

Here's what happens the moment someone opens a GitHub issue:

🔍 **Step 1 — AI reads the issue**
GPT-4o-mini analyzes the title + description and assigns a severity:
— P0 = System down / data loss
— P1 = Major feature broken
— P2 = Workaround exists
— P3 = Minor / cosmetic

⚡ **Step 2 — Auto-labels the issue**
Severity label + area label (frontend / backend / devops / security) added automatically.

👤 **Step 3 — Assigns the right team**
No more "who should handle this?" — the AI routes it to the right person instantly.

💬 **Step 4 — Posts a triage comment**
A formatted summary appears on the issue itself — severity reason, assignment reason, plain-English summary.

🚨 **Step 5 — Fires Slack alert for P0/P1 only**
Critical issues get a red alert in #engineering-alerts with all context included.

The whole thing runs in under 5 seconds.

Your team wakes up to triaged, labeled, assigned issues — not a wall of unread noise.
