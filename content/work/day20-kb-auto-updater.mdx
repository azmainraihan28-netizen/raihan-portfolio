## The Problem

1. Support teams manually update KB articles — wastes 2–4 hours per article; documentation is always 3–6 months behind the product
2. Customers read outdated docs and file repeat tickets — a self-reinforcing cycle that grows support load over time
3. No system to track which articles need updating based on ticket volume or new product features
4. KB management tools (Guru, Tettra, Document360) cost $100–$500/month — plus human time to write the content
5. New support agents have nothing useful to learn from — stale KB, patchy coverage, and slow onboarding

## The Solution

A webhook-triggered n8n workflow that:
- Receives any resolved support ticket via POST (compatible with Zendesk, Freshdesk, HelpDesk, or manual POST)
- Uses GPT-4o-mini to decide: Create new article / Update existing article / Skip (too specific) — with a 1–10 confidence score
- Generates a full markdown KB article: intro, problem, step-by-step solution, tips, related topics (300–500 words)
- Creates a new Notion page or updates an existing one automatically via Notion API
- Logs every action to Google Sheets (action type, confidence, article metadata, skip reason)
- Notifies the team in Slack with a preview of the drafted article

## How It Works

Node-by-node flow:

- **Webhook** — POST /kb-auto-updater
- **Set** — Normalizes 11 input fields with safe defaults
- **IF** — Validates required fields (ticket_title + resolution) → 400 error on missing
- **OpenAI (gpt-4o-mini)** — Decides Create/Update/Skip + drafts full article as JSON
- **Code** — Safely parses AI JSON with full fallback on parse error
- **IF (×2)** — Routes to Create path, Update path, or Skip path
- **HTTP Request (Notion API ×2)** — POST to create new page / PATCH to update existing page
- **No Op (×2)** — Silent passes for skip and non-update paths
- **Google Sheets** — 15-column KB action log
- **Slack** — Team notification with article preview and action taken
- **Respond to Webhook (×2)** — Returns 200 success JSON or 400 validation error
